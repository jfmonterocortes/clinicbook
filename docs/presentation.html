Background

The Problem

  • Small clinics rely on phone calls and paper to manage appointments
  • No access control over sensitive patient information
  • No structured way to share medical documents securely
  • Three different types of users with different needs
  • Patients - need to book appointments and share documents
  • Doctors - need to manage their schedule and review patient files
  • Admins - need to approve accounts and monitor the system
Project

ClinicBook

  • Role-based web application with realistic security controls
  • Patients: search doctors, book appointments, upload documents
  • Doctors: manage schedule, book for patients, download documents
  • Admins: approve accounts, book appointments, monitor audit log
  • Doctor approval workflow - new doctors wait for admin approval
  • Backend: Node.js + Express
  • Database: SQLite (better-sqlite3)
  • Frontend: HTML + Bootstrap 5 + JavaScript
  • Security: bcrypt, helmet, cookie-session, express-validator, multer, file-type
Design

Application Architecture

Front-End Layer

HTML + Bootstrap 5 + JavaScript  ·  Communicates only via /api/*  ·  No direct database access

↓ HTTP / JSON

Each entry point is a realistic attack vector - the following slides cover the controls applied to each one.

Security Controls

Authentication

  • Passwords hashed with bcrypt at cost factor 12 - plain-text passwords never stored
  • Session cleared on every login to prevent session fixation attacks
  • Cookie flags: httpOnly (JS can't read it), sameSite=lax (CSRF protection), secure in production
  • Sessions expire after 2 hours
  • Timing attack prevention - bcrypt always runs even for unknown emails using a pre-computed dummy hash, so attackers can't tell if an email is registered based on response speed
Security Controls

Authorization

  • requireAuth middleware on every protected route - checks for a valid session before any handler code runs
  • requireRole middleware for role-specific routes - returns 403 if the user's role does not match
  • Ownership checks at the data level - queries include AND uploader_id = ? so guessing another user's ID returns nothing (prevents IDOR attacks)
  • Doctor approval workflow - is_approved = 0 on registration, requireApproved blocks access until admin approves
  • Admins and doctors can book appointments on behalf of patients - role enforces what each can do
Security Controls

Input Validation & SQL Injection Prevention

Input Validation

  • All inputs validated server-side with express-validator - client-side is not trusted
  • Email format, password complexity (8+ chars, uppercase, number)
  • Length limits on all text fields
  • Future date validation on appointments
  • Integer range checks on all ID parameters

SQL Injection Prevention

  • Every query uses prepared statements - user input is never concatenated into SQL
  • Input like ' OR '1'='1 is treated as a literal string, not SQL
  • Returns zero results instead of dumping the database
Security Controls

File Upload Security

  • Layer 1 - MIME type whitelist: only application/pdf, image/png, image/jpeg accepted
  • Layer 2 - File extension whitelist: only .pdf, .png, .jpg, .jpeg accepted
  • Layer 3 - Magic bytes check: reads the first bytes of the actual file and verifies the signature matches - catches renamed files (e.g. a PHP script renamed to .jpg)
  • Size limit: 2 MB maximum enforced by the server
  • UUID filenames: original filename is never used on disk - stored under a random UUID to prevent path traversal
  • Stored outside public/: files are not directly web-accessible, all downloads go through an authenticated API endpoint
Security Controls

Additional Controls

  • Rate Limiting: 10 login attempts per IP per 5 minutes - blocks brute force attacks
  • Security Headers: helmet middleware sets CSP, X-Frame-Options, HSTS, X-Content-Type-Options on every response
  • Error Handling: all errors return a generic message - stack traces and internal details are never sent to the client
  • Secrets: SESSION_SECRET stored in .env, gitignored - never hardcoded in source code
  • Audit Logging: every sensitive action (login, logout, booking, upload, approval) is logged to the audit_logs table with IP and timestamp

Summary

Conclusion

  • Security designed in from the start, not added at the end
