  • No structured way to share medical documents securely
  • Scheduling conflicts and lost patient records are common

Project

ClinicBook

A role-based web application with a clear client-server architecture and realistic security controls built for the course.

  • Search for doctors by name or specialty
  • Book and manage appointments
  • Upload medical documents securely (PDF, PNG, JPG - max 2 MB)
  • Role-based dashboards for Patient, Doctor, and Admin
  • Doctor approval workflow and full audit log

HTTP / JSON

Routes: /api/login  /api/register  /api/doctors  /api/appointments  /api/uploads  /api/admin/*

Each entry point is a realistic attack vector. The following slides detail the security controls applied to each one.

Security Controls

Authentication

🔑
Password Hashing bcrypt with cost factor 12 - plain-text passwords never stored in the database
🔄
Session Fixation Prevention req.session.regenerate() called on every successful login - prevents attacker-set session IDs
🍪
Secure Cookie Flags httpOnly (JS can't read it), sameSite=lax (CSRF), secure in production - 2 hour expiry
⏱️
Timing Attack Prevention bcrypt always runs even for unknown emails using a pre-computed dummy hash
// bcrypt always runs - prevents timing leak const hash = user ? user.password_hash : DUMMY_HASH; const match = await bcrypt.compare(password, hash); // Regenerate session ID on login req.session.regenerate(async (err) => { req.session.user = { id, email, role }; });
Why timing matters

If bcrypt is skipped for unknown emails, the server responds faster. An attacker can use this timing difference to enumerate registered email addresses.

Security Controls

Authorization

🛡️
requireAuth Middleware Every protected route checks for a valid session before any handler code runs
👥
Role-Based Access Control requireRole('admin') / requireRole('doctor','admin') on all role-specific routes - returns 403
🔒
IDOR Prevention Ownership checks in every query: WHERE id = ? AND uploader_id = ? - guessing another user's ID returns nothing
✅
Doctor Approval Workflow is_approved = 0 on registration. requireApproved middleware blocks all access until admin manually approves
// Route-level role check router.get('/users', requireAuth, requireApproved, requireRole('admin'), handler ); // Data-level ownership check (IDOR) const doc = db.prepare(` SELECT * FROM documents WHERE id = ? AND uploader_id = ?` ).get(docId, userId);
Without ownership checks

A patient could call /api/uploads/2/download and access any other patient's medical file just by incrementing the ID.

Security Controls

Input Validation & SQL Injection

Input Validation (express-validator)
  • All inputs validated server-side - client-side validation is not trusted
  • Email format checking
  • Password complexity: 8+ chars, one uppercase, one number
  • Length limits on all text fields
  • ISO 8601 date format + must be a future date for appointments
  • Integer range checks on all ID parameters
Client-side bypass

Anyone can skip the browser and send a request directly with Postman or curl. Server-side validation is mandatory.

SQL Injection Prevention
// All queries use prepared statements sql += ` AND u.full_name LIKE ?`; params.push(`%${name}%`); db.prepare(sql).all(...params); // Attack: ' OR '1'='1 // Result: treated as literal string // No data leaked - returns 0 results
Why prepared statements work

The database driver separates the query structure from the data. User input can never change the SQL logic - it is always treated as data.

Security Controls

File Upload Security

Three Layers of Validation
1
MIME Type Whitelist application/pdf  image/png  image/jpeg
2
File Extension Whitelist .pdf  .png  .jpg  .jpeg
3
Magic Bytes Verification Reads first 4100 bytes of actual file content using file-type library
Why 3 layers?

Renaming script.php to image.jpg fools a single check. Magic bytes verification detects disguised files regardless of extension or MIME header.

Additional Protections
  • 2 MB maximum file size enforced by multer
  • Files saved with UUID names - original filename never used on disk
  • uploads/ stored outside public/ - not directly accessible via browser
  • Downloads only via authenticated route with ownership check
// Magic bytes check const detected = await fileTypeFromBuffer(buf); if (!detected || !ALLOWED_MIMES.includes(detected.mime)) { fs.unlinkSync(req.file.path); // remove file return res.status(400).json({ error: ... }); }
Security Controls

Additional Controls

🚦
Rate Limiting 10 login attempts per IP per 5 minutes. 5 registrations per IP per hour. Returns 429 when exceeded.
🪖
Security Headers (helmet) CSP, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, HSTS, Referrer-Policy: no-referrer
🚫
Error Handling Global handler returns only "An unexpected error occurred" - no stack traces or file paths sent to client
🔐
Secrets Management SESSION_SECRET in .env, excluded via .gitignore. .env.example committed with placeholder values.
📋
Audit Logging Every sensitive action written to audit_logs with user ID, IP, user agent, timestamp. Events: REGISTER, LOGIN_SUCCESS, LOGIN_FAIL, LOGIN_BLOCKED, LOGOUT, APPOINTMENT_BOOKED, APPOINTMENT_CANCELLED, DOCUMENT_UPLOADED, DOCUMENT_DOWNLOADED, USER_APPROVED, USER_SUSPENDED

ThreatExampleMitigation

Reflection

Limitations & Future Work

Current Limitation

No encryption at rest

  • Database file and uploads/ folder are not encrypted on disk
  • Passwords are safe - bcrypt protects them even if the file leaks
  • Other data (emails, names, appointment notes) would be readable if the server is compromised

Documented honestly in the report rather than left unaddressed.

Future Improvements
  • Disk-level encryption on the production server
  • Encrypted cloud storage for uploaded files
  • Email notifications for appointment confirmations
  • HTTPS with a proper TLS certificate
  • Automated test suite for security controls
  • Shorter session windows with refresh tokens
Summary

Conclusion

  • All 5 required entry points implemented and secured
  • 6 OWASP Top 10 categories addressed with real code controls
  • STRIDE threat model applied to every entry point
  • Security designed in from the start, not added at the end
  • Limitations documented honestly rather than hidden
